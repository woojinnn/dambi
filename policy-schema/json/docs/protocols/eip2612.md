# eip2612

## 개요
- **그룹**: `signature` (EIP-712 서명 어댑터)
- **chain**: 토큰별 (verifying_contract = ERC-20 토큰 컨트랙트 주소)
- **출처**: EIP-2612 표준 (각 토큰 컨트랙트 자체 구현)
- **emit**: `Action::Eip2612(Eip2612Action)` → Cedar `Action::"signature.eip2612"`

## primary type

| primary_type | 매칭 |
|---|---|
| `Permit` | case-insensitive |

EIP-2612 메시지 구조:
```solidity
struct Permit {
  address owner;
  address spender;
  uint256 value;
  uint256 nonce;
  uint256 deadline;
}
```

## Eip2612Action 14필드

| 필드 | 정책 활용 |
|---|---|
| `signer` (= owner 강제 일치) | actor 식별 |
| `spender` | spender allowlist |
| `is_unlimited` (value == uint256 max) | unlimited 차단 (`no-unlimited-amount.cedar`) |
| `nonce_valid` | nonce sanity (`nonce-sanity.cedar`) |
| `deadline` | deadline cap (`deadline-le-1h.cedar`) |
| `value` | human cap (`max-human-value-50.cedar`) |
| `total_approved_usd` | USD cap (`max-usd-100.cedar`) |
| `chain_id` vs `domain_chain_id` | chain 일치 |

## 토큰 화이트리스트

verifying_contract 는 토큰 컨트랙트 자체이므로, 어댑터의 `TokenLookup` 이 자동으로 화이트리스트 역할:
- `TokenLookup` 에 등록된 토큰만 EIP-2612 어댑터가 매칭
- 미등록 토큰은 매칭 실패 → 파이프라인이 `Eip712OtherAction::from_request()` 합성 (catch-all)

## owner = signer 강제

어댑터가 `message.owner != sig.signer` 인 요청을 빌드 단계에서 거부 (`AdapterError`). 이는 host 가 owner 를 잘못 제공했거나 dApp 이 다른 owner 를 claim 하는 phishing 일부를 차단하기 위함.

ERC-1271 / smart-wallet flow (ECDSA key 가 owner 와 다른 경우) 에서는 host 가 이미 onchain owner 를 resolve 해서 `signer` 로 제공해야 함.

## 특이사항
- 어댑터는 ECDSA 서명 검증을 하지 않음.
- DAI 같은 비표준 EIP-2612 변형 (`PERMIT_TYPEHASH` 가 다름, allowed bool 사용) 은 v0.1 범위 외 — `Eip712Other` catch-all 로 떨어짐.
